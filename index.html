Senior Case Studies

Structured using a real-world workflow: signal → triage → diagnosis → fix → prevention.

SEV-1: Merchant-impacting incident (end-to-end ownership)

  • Signal: Multiple SEV-1 alerts triggered across Payment, Authentication, and Reporting services.
  • Triage: Dynatrace Smartscape analysis showed all impacted services were connected to the same database.
  • Diagnosis: Splunk logs revealed a recently modified archival job was creating heavy DB load and exhausting connection pools.
  • Fix: Paused the archival job and adjusted DB connection limits to stabilize services.
  • Prevention: Redesigned the archival process to run in smaller batches and implemented DB resource controls.

Result: Services were restored quickly and future database impact was prevented.

Role-Based Monitoring Dashboard

  • Signal: Business needed a faster, role-based dashboard for better credit visibility.
  • Triage: Worked with BA and product teams to define role-based data needs and performance expectations.
  • Diagnosis: Identified APIs were over-fetching data and missing SQL indexes were slowing filter performance.
  • Fix: Built optimized .NET Core APIs and enhanced Angular UI with server-side pagination.
  • Prevention: Introduced API versioning and performance baselines.

Result: Dashboard load time improved by ~45%.

Microservices Reporting Optimization Using SQL Materialized View

  • Signal: Reporting was slow due to real-time aggregation of large transactional data.
  • Triage: Analyzed latency and aligned with stakeholders on performance targets.
  • Diagnosis: Identified heavy SQL joins occurring across microservices.
  • Fix: Built a reporting microservice using a SQL Server indexed (materialized) view.
  • Prevention: Added a refresh strategy and architectural guidelines for read-heavy workloads.

Result: Report generation time reduced by ~60% and database load decreased.

Database: Deadlocks / blocking causing API timeouts (SQL Server / Oracle)

  • Signal: Sudden increase in API latency + timeout errors; elevated DB CPU/locks during peak hours.
  • Triage: Reduced blast radius (temporary throttling / feature flag / queueing) and stabilized user-facing flows.
  • Diagnosis: Identified blocking chains and deadlock patterns using DB monitoring and correlated Splunk traces to isolate queries/endpoints.
  • Fix: Refactored queries/stored procedures, added targeted indexes, reduced transaction scope, and introduced safe retry for transient failures.
  • Prevention: Added DB guardrails (timeouts/thresholds), dashboards for waits/latency, and a lock escalation runbook.
