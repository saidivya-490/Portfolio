Senior Case Studies

Written in the format hiring managers look for: signal → diagnosis → decision → fix → prevention.

SEV-1: Merchant-impacting incident (end-to-end ownership)

  • Signal: spike in errors + monitoring alerts.
  • Triage: stabilize service and communicate impact & timeline.
  • Diagnosis: Splunk transaction tracing + APM call graphs + DB checks.
  • Fix: safe patch/DB correction with controlled rollout and validation.
  • Prevention: tuned alerts, dashboards, and runbook updates.

Batch operations: Autosys job failures within SLA

  • Problem: job failures impacting downstream workflows.
  • Approach: correlate logs + dependency health + DB behavior.
  • Resolution: fix root cause (timeouts/retries/query tuning) and verify recovery.
  • Hardening: better alerting, reduced noise, clearer escalation path.

Database: Deadlocks / blocking causing API timeouts (SQL Server / Oracle)

  • Signal: Sudden increase in API latency + timeout errors; elevated DB CPU/locks during peak hours.
  • Triage: Reduced blast radius (temporary throttling / feature flag / queueing) and stabilized user-facing flows.
  • Diagnosis: Identified blocking chains and deadlock patterns using DB monitoring (locks, wait stats), correlated with Splunk traces to isolate the exact API endpoints and queries involved.
  • Decision: Prioritized a safe fix path: reduce lock duration first (query rewrite + indexing), then adjust transaction scope and isolation where needed.
  • Fix: Refactored the stored procedure/query to avoid wide scans, added targeted indexes, reduced transaction scope, and introduced idempotent retry for safe transient failures.
  • Prevention: Added DB guardrails (timeouts, alert thresholds), created a runbook for lock escalation, and added dashboards tracking waits/latency/error rates.

Data integrity: Preventing duplicate/incorrect records in distributed workflows

  • Signal: Intermittent duplicates / mismatched counts reported by business; issues spiked during retries or partial outages.
  • Triage: Identified impacted time window, paused risky replays, and used reconciliation queries to quantify impact.
  • Diagnosis: Root cause was a non-idempotent write path + retry behavior during transient failures (network/timeout), leading to duplicate inserts.
  • Decision: Implemented an idempotency strategy to make retries safe without slowing normal throughput.
  • Fix: Added a deterministic idempotency key, enforced uniqueness at the DB level, and updated the API/service layer to use upsert/merge patterns where appropriate.
  • Prevention: Added reconciliation job/runbook, alerting for anomaly detection (spikes in duplicates), and expanded test coverage for retry + failure scenarios.
